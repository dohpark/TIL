## 프로토타입

- 자바스크립트를 이루고 있는 거의 모든 것이 객체다!
- 자바스크립트는 프로토타입 기반의 객체지향 프로그래밍 언어다!
- es6부터 클래스를 도입했는데, 클래스와 생성자 함수 모두 프로토타입 기반의 인스턴스를 생성하지만 동일하지 않다.
- 클래스는 생성자 함수보다 엄격하며, 생성자 함수가 없는 기능도 있다

### 객체지향 프로그래밍

- 객체지향 프로그래밍은 객체의 집합으로 프로그램을 표현하려는 프로그래밍 패러다임임.
- 실체를 여러 속성들로 추상화한 값들을 하나의 단위로 구성한 자료구조를 객체라고 함.
- 객체는 상태(프로퍼티)와 동작(메서드)의 구성되며, 이를 하나의 논리적 단위로 묶을려고 함.
- 객체는 다른 객체와 관계성을 가질 수 있음. 다른 객체와 함께 데이터 처리할 수도 있음.

### 상속과 프로토타입

- 상속이란 어떤 객체의 프로퍼티 or 메서드를 다른 객체가 상속받아 그대로 사용할 수 있음을 뜻함.
- 자바스크립트는 프로토타입을 기반으로 상속을 구현하여 기존의 코드를 재사용하는 방식으로 불필요한 중복을 제거함.
- 생성자 함수에서 동일한 내용의 메서드를 단 하나만 생성하여 인스턴스가 상속받아 사용하는 것이 코드의 재활용 측면에서 더 긍정적임.
- 자바스크립트는 프로토타입을 기반으로 상속을 구현함. 생성자 함수로 생성한 인스턴스는 해당 생성자 함수의 프로토타입의 모든 프로퍼티와 메서드를 상속받음.
- 즉 인스턴스가 상위 객체인 프로토타입의 프로퍼티와 메서드를 상속받아 사용가능함.

### 프로토타입 객체

- 프로토타입 객체는 객체 간 상속을 구현하기 위해 사용함.
- 모든 객체는 `[[prototype]]`이라는 내부 슬롯을 가진다. 여기에 저장되는 프로토타입은 객체 생성방식에 의해 결정됨.
- 객체 리터럴로 생성된 객체의 프로토타입은 Object.prototype이며, 생성자 함수의 경우 해당 생성자 함수의 prototype 프로퍼티에 바인딩 되어 있는 객체임.
- 모든 객체는 하나의 프로토타입을 가지며, 모든 프로토타입은 생성자 함수와 연결되어 있음. 즉 객체와 프로토타입과 생성자 함수는 서로 연결되어 있음.

- `__proto__` 접근자 프로퍼티

  - 모든 객체는 `__proto__`접근자 프로퍼티를 통해 자신의 프로토타입(`[[prototype]]` 내부 슬롯)에 간적접으로 접근 가능함.

  - `__proto__`는 접근자 프로퍼티이다.
    - `__proto__` 접근자는 접근자 함수이기에 `[[get]]`, `[[set]]` 프로퍼티 어트리뷰트로 구성되어 있어 두 getter, setter 함수를 사용할 수 있음.
  - `__proto__` 접근자 프로퍼티는 상속을 통해 사용된다.
    - `__proto__` 접근자는 Object.prototype의 프로퍼티이며, 이를 상속을 통해 `Object.prototype.__proto__` 접근자 프로퍼티를 사용함.
  - `__proto__`접근자 프로퍼티를 통해 프로토타입에 접근하는 이유
    - 프로토타입에 접근하기 위해 접근자 프로퍼티를 사용하는 이유는 상호 참조에 의해 프로토타입 체인이 생성하는 것을 방지하기 위함임.
    - 서로가 자신의 프로토타입이 되는 비정상적인 프로토타입 체인이 만들어지는 것을 방지하기 위함. (프로토타입체인은 단방향 링크드 리스트여야함.)
    - 순환 참조하는 프로토타입 체인이 생성되면 프로퍼티 검색 시 무한루프에 빠질 수 있음.
  - `__proto__`접근자 프로퍼티를 코드 내에서 직접 사용하는 것을 권장하지 않음
    - `__proto__` 접근자 프로퍼티 사용을 권장하지 않음.
    - 직접 상속을 통해 Object.prototype을 상속받지 않는 경우 `__proto__`을 사용못할 경우가 있음
    - 참조 취둑에는 Object.getPrototypeOf 메서드를 권장
    - 교체시에는 Object.setPrototypeOf 메서드 사용을 권장함

- 함수 객체의 prototype 프로퍼티

  - 함수 객체만이 소유하는 prototype 프로퍼티는 생성자 함수가 생성할 인스턴스의 프로토타입을 가리킴.
  - non-constructor은 prototype 프로퍼티를 소유하지 않으며 프로토타입도 생성하지 않음.
  - 생성자 함수로 호출하지 않은 일반함수도 prototype 프로퍼티를 갖고 있지만 큰 의미는 없다
  - 모든 객체가 지니는 `__proto__`접근자 프로퍼티와 함수 객체만의 prototype 프로퍼티는 결국엔 동일한 프로토타입을 가리킴. 하지만 사용하는 주체가 다름.
  - `__proto__`접근자 프로퍼티는 모든 객체가 소유하며 자신의 프로토타입에 접근 또는 교체하기 위해 사용됨.(사용 주체: 모든 객체)
  - prototype 프로퍼티는 constructor가 소유하며 생성자 함수가 자신이 생성할 인스턴스의 프로토타입을 할당하기 위해 사용함 (사용주체: 생성자 함수)

- 프로토타입의 constructor 프로퍼티와 생성자 함수

  - 모든 프로토타입은 constructor 프로퍼티를 가지며, 자신을 참조하고 있는 생성자 함수를 가리킴.
  - 인스턴스 객체는 constructor을 프로퍼티를 통해 생성자 함수와도 연결이 가능함. 왜냐하면 해당 객체의 프로토타입이 이를 지니며 상속하기 때문임.

### 리터럴 표기법에 의해 생성된 객체의 생성자 함수와 프로토타입

- 리터럴 표기법에 의해 생성된 객체의 constructor 프로퍼티는 Object 생성자 함수임.
- 그런데 ECMAScript 사양을 보면 Object 생성자 함수 호출과 객체 리터럴의 평가는 다르게 정의되어있음(물론 일부 내용은 동일하지만). 그럼 어떻게 이어진거임?
- 리터럴 표기법에 의해 생성된 객체는 상속을 위한 프로토타입이 필요하기에 가상적인 생성자 함수를 가지고 있음.
- 프로토타입과 생성자 함수는 단독으로 존재할 수 없고 언제나 쌍으로 존재해야함.
- 리터럴 표기법에 의해 생성된 객체는 생성자 함수에 의해 생성된 객체는 아니지만 본질적인 면에서 큰 차이는 없음.
- 프로토타입의 constructor 프로퍼티를 통해 연결되어 있는 생성자 함수를 리터럴 표기법으로 생성한 객체를 생성한 생성자 함수로 생각해도 큰 무리는 없음.

### 프로토타입의 생성 시점

- 프로토타입은 생성자 함수가 생성될 때 같이 생성됨. (둘이 쌍으로 존재하기 때문)
- 사용자 정의 생성자 함수와 빌트인 생성자 함수로 나눠서 봅세

- 사용자 정의 생성자 함수와 프로토타입 생성시점

  - constructor만 생성자 함수로서 호출할 수 있는데, 함수 객체 생성 시점에 프로토타입도 같이 생성됨.
  - 함수 선언문은 런타임 이전에 먼저 실행되어 함수 객체를 생성하는데, 이때 프로토타입도 같이 생성되어 생성자 함수의 prototype 프로퍼티에 바인딩됨.
  - 프로토타입도 프로토타입을 갖게 되는데, 생성된 프로토타입의 프로토타입은 Object.prototype임.

- 빌트인 생성자 함수와 프로토타입 생성시점

  - 빌트인 생성자 함수가 생성되는 시점에 프로토타입도 생성됨.
  - 빌트인 생성자 함수는 전역 객체가 생성되는 시점에 생성됨. 이 때 프로토타입도 생성되어 빌트인 생성자 함수의 prototype 프로퍼티에 바인딩됨.
  - 전역 객체는 코드가 생되기 이전 단계에서 JS엔진에 의해 생성됨. 전역 객체는 표준 빌트인 객체들과 호스트 객체, 그리고 var 키워드로 선언한 전역 변수와 전역 함수를 프로퍼티로 가짐. Object도 전역 객체의 프로퍼티이며, 전역 객체가 생성되는 시점에 생성됨.
  - 그렇기 때문에 객체가 생성되기 이전에 생성자 함수와 프로토타입은 이미 객체화 되어 존재함.
  - 생성자 함수 또는 리터럴 표기법으로 객체를 생성하면 프로토타입은 생성된 객체의 `[[prototype]]` 내부 슬롯에 할당됨.

### 객체 생성 방식과 프로토타입의 결정

- 객체를 생성하는 방법은 객체 리터럴, Object 생성자 함수, 생성자 함수, Object.create 메서드, 클래스가 있음
- 위 객체 모두 OrdinaryObjectCreate에 의해 생성된다는 공통점이 있음
- 추상 연산 OrdinaryObjectCreate는 자신이 생성할 객체의 프로토타입을 인수로 전달 받음. 그리고 인수로 받은 프로토타입을 `[[prototype]]` 내부 슬롯에 할당한 다음 생성한 객체를 반환함.
- 즉 프로토타입은 추상연산 OrdinaryObjectCreate에 전달되는 인수에 의해 결정됨. 이 인수는 객체 생성 방식에 의해 결정됨.

- 객체 리터럴에 의해 생성된 객체의 프로토타입

  - 객체 리터럴에 의해 생성할 때 추상연산 OrdinaryObjectCreate에 전달되는 프로토타입은 Object.prototype임.
  - 그 덕분에 Object.prototype의 constructor 및 여러 메서드를 자유롭게 상속받아 사용 가능함.

- Object 생성자 함수에 의해 생성된 객체의 프로토타입

  - 마찬가지로 추상연산 OrdinaryObjectCreate에 전달되는 프로토타입은 Object.prototype임.
  - 객체 리터럴과 Object 생성자 함수에 의한 객체 생성 방식의 차이점은 프로퍼티를 추가하는 방식에서 차이가 있음.
  - 객체 리터럴 방식은 객체 리터럴 내부에 프로퍼티를 추가함.
  - Object 생성자 함수 방식은 빈 객체를 생성한 이후, 프로퍼티를 추가함.

- 생성자 함수에 의해 생성된 객체의 프로토타입

  - 추상연산 OrdinaryObjectCreate에 전달되는 프로토타입은 생성자 함수의 prototype 프로퍼티에 바인딩되어 있는 객체임.
  - 이를 통해 생성자 함수, 생성자 함수의 prototype 프로퍼티에 바인딩되어 있는 객체, 생성된 객체 사이에 연결이 만들어짐.
  - 프로토타입에 프로퍼티를 추가/삭제할 수 있고 프로토타입 체인에 바로 반영됨.

### 프로토타입 체인

- 자바스크립트는 객체의 프로퍼티에 접근할려 할때 프로퍼티가 없다면 `[[prototype]]` 내부 슬롯의 참조를 따라 프로토타입의 프로퍼티를 순차적으로 검색하는데 이를 프로토타입 체인이라고 함.
- 프로토타입 체인은 JS가 객체지향 프로그래밍의 상속을 구현하는 메커니즘임.
- 프로토타입 체인의 최상위에 위치하는 객체는 Object.prototype이기에 모든 객체는 Object.prototype을 상속받음.
- 그렇기에 Object.prototype을 프로토타입 체인의 종점이라고 함. (Object.prototype의 프로토타입은 null임)
- 프로토타입 체인의 종좀에서도 프로퍼티를 못 찾으면 undefined를 반환함.
- 이렇게 계층적인 구조에서 객체의 프로퍼티를 검색할 수 있으니 프로토타입 체인은 상속과 프로퍼티 검색을 위한 메커니즘이라고 볼 수도 있음.
- 스코프체인은 식별자 검색을 위한 메커니즘. 하지만 서로 협력하여 식별자와 프로퍼티를 검색하는데 사용됨.

### 오버라이딩과 프로퍼티 새도잉

- 오버라이딩은 상위 프로토타입이 지니는 메서드를 하위 프로토타입이 재정의하여 사용하는 것
- 프로퍼티 새도잉은 이런 상속 관계에서 프로퍼티가 가려지는 현상임.
- 하위 객체를 통해 프로토타입의 프로퍼티를 변경 or 삭제는 불가능함. get 엑세스는 허용되나 set 엑세스는 허용되지 않음.
- 프로토타입 체인이 아닌 프로토타입에 직접 접근해야 삭제 또는 변경이 가능함.

### 프로토타입의 교체

- 프로토타입은 동적으로 변경할 수 있기에 상속관계를 변경할 수 있다.

- 생성자 함수에 의한 프로토타입 교체

  - 생성자 함수가 생성할 객체의 프로토타입을 객체 리터럴로 교체함.
  - 교체한 객체 리터럴에 constructor 프로퍼티가 없으면 그 프로토타입 체인을 타고 올라가 Object.prototype의 constructor을 가르키게 됨
  - 교체한 객체 리터럴에 constructor도 명시하면 문제 해결.

- 인스턴스에 의한 프로토타입 교체

  - `__proto__` 접근자 프로퍼티를 통해 접근하기에 이미 생성된 객체의 프로토타입을 교체하는 것임.
  - 그래서 생성자 함수에 의한 프로토타입 교체는, 생성자 함수의 prototype 프로퍼티가 교체된 프로토타입을 가리킴.
  - 그 반면에 인스턴스에 의한 프로토타입 교체는, 생성자 함수의 prototype 프로퍼티가 교체된 프로토타입을 가리키지 않음.
  - 그러므로 생성자 함수의 prototype 프로퍼티와 프로토타입 간의 연결도 설정해야하고, 인스턴스 객체의 프로토타입도 새로 설정해야함.
  - 매우 번거롭게 되니 직접 교체하지 말고 직접 상속을 통해 상속관계를 인윚적으로 설정하거나, 클래스를 사용하는 것이 더 편함.

### instanceof 연산자

- 우변의 생성자 함수의 prototype에 바인딩된 객체가 좌변의 객체의 프로토타입 체인 상에 존재하면 true 아니면 false.
- constructor 프로퍼티와 생성자 함수 간의 연결이 파괴되어도 instanceof는 영향을 받지 않음.

### 직접 상속

- Object.create에 의한 직접 상속

  - Object.create 메서드를 통해 프로토타입을 지정하여 새로운 객체를 생성할 수 잇음. 이 때 추상 연산 OrdinaryObjectCreate를 호출함.
  - 첫번째 매개변수에는 프로토타입으로 지정할 객체를 전달함.
  - 두번째 매개변수에는 생성할 객체의 프로퍼티 키와 프로퍼티 디스크립터 객체로 이뤄진 객체를 전달함. (옵션임)
  - Object.create 장점
    - 1 new 연산자가 없어도 객체 생성가능
    - 2 프로토타입을 지정하면서 객체를 생성할 수 잇음
    - 3 객체 리터럴에 의해 생성된 객체도 상속 받을 수 있음
  - ESLint는 Object.prototype 빌트인 메서드를 객체가 직접 호출하는 것을 권장하지 않음.
  - 그 이유는 Object.create 메서드를 통해 프로토타입 체인의 종점에 위치하는 객체를 생성할 수 있기 때문임. 프로토타입 종점에 위치하는 객체는 Object.prototype의 빌트인 객체를 사용할 수 없음.
  - 그렇기에 Function.prototype.apply/call/bind 메서드에 의한 간접적으로 호출하는 것을 권장함.

- 객체 리터럴 내부에서 `__proto__`에 의한 직접 상속

  - Object.create 메서드의 두번째 인자로 프로퍼티 정의하는것은 번거로움.
  - es6부터 객체 리터럴 내부에서 `__proto__` 접근자 프로퍼티를 사용하여 직접 상속을 구현할 수 있음.

### 정적 프로퍼티/메서드

- 정적 프로퍼티/메서드는 생성자 함수로 인스턴스를 생성하지 않아도 참조/호출할 수 있는 프로퍼티/메서드를 말함.
- 정적 프로퍼티/메서드는 생ㅅ어자 함수가 생ㅅ어한 인스턴스로 참조/호출할 수가 없음.
- 정적 프로퍼티/메서드는 인스턴스의 프로토타입 체인에 속한 객체의 프로퍼티/메서드가 아니므로 인스턴스로 접근할 수가 없음.
- 프로토타입 메서드를 호출하기 위해서는 인스턴스를 생성해야하지만, 정적 메서드는 인스턴스를 생성하지 않아도 호출할 수 있음.
- MDN 보면 정적 프로퍼티/메서드와 프로토타입 프로퍼티/메서드 구분하여 소개함.
- 프로토타입 프로퍼티/메서드를 표기할 때 prototype을 #으로 표기하는 경우도 있음.

### 프로퍼티 존재 확인

- in 연산자

  - key(프로퍼티 키를 나타내는 문자열) in object(객체로 평가되는 표현식)
  - 확인 대상 객체의 프로퍼티뿐만 아니라 확인 대상 객체가 상속받은 모든 프로토타입의 프로퍼티를 확인하니 주의해서 사용해야함.
  - Reflect.has 메서드도 사용할 수 있음. in 연산자와 동일함. ex. Reflect.has(person, 'toString')

- Object.prototype.hasOwnProperty 메서드
  - Object.prototype.hasOwnProperty 메서드를 사용하면 객체에 특정 프로퍼티가 존재하는지 알 수 있음.
  - 고유의 프로퍼티 키만 true를 반환하고 상속받은 프로토타입의 프로퍼티 키인 경우 false를 반환

### 프로퍼티 열거

- for... in 문

  - 객체의 모든 프로퍼티를 순회하며 열거하려면 for ...in 문을 사용함
  - for (변수선언문 in 객체) { ... }
  - for... in 문은 객체의 프로퍼티 개수만큼 순회하면 프로퍼티 키를 할당함.
  - 특징1 상속받은 프로토타입의 프로퍼티까지 열거. toString 열거하지 않은 이유는 프로퍼티 어트리뷰트 `[[enumberable]]` 값이 false이기 때문임.
  - 다시 정의. for... in 문은 프로토타입 체인 상에 존재하는 모든 프로토타입의 프로퍼티 중에서 프로퍼티 어트리뷰트 `[[enumerable]]`의 값이 true인 프로퍼티를 순회하며 열거함. 객체 자신의 프로퍼티만 열거하려면 Object.prototype.hasOwnProperty 메서드를 통해 객체 자신의 프로퍼티인지 확인해야함.
  - 특징2 프로퍼티 키가 심벌인 프로퍼티는 열거하지 않음.
  - 특징3 프로퍼티를 열거할 때 순서를 보장하지 않음. 대부분의 모던 브라우저는 순서를 보장하고 숫자인 프로퍼티 키에 대해서는 정렬을 실시함.
  - 배열에는 for문, for... of 문, Array.prototype.forEach 메서드를 사용하기를 권장함.(forEach는 요소가 아닌 프로퍼티는 제외함.)

- Object.keys/values/entries 메서드

  - 객체 자신의 고유 프로퍼티만 열거하기 위해서는 Object.keys/values/entries 메서드를 권장함
  - Object.keys는 객체 자신의 열거 가능한 프로퍼티 키를 배열로 반환함.
  - Object.values는 객체 자신의 열거 가능한 프로퍼티 값을 배열로 반환함.
  - Object.entries는 객체 자신의 열거 가능한 프로퍼티 키와 값의 쌍의 배열을 배열에 담아 반환함.
